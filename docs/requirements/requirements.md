# メモアプリ 要件定義（v0.3）

## 1. 背景・課題

| # | 課題 | 原因 |
|---|---|---|
| ① | 1つのメモ内で修正箇所を探すのに何度もスクロールする | メモが長文化・肥大化しやすい |
| ② | メモの数が増えると、どのメモに何を書いたか分からなくなる | メモ単位の整理・横断検索手段が弱い |
| ③ | 長文メモは後から見返しにくい | 内容が構造化されず書きっぱなしになる |
| ④（根本原因） | 1つのテーマの中に複数の異なる観点（一覧・手順・今後やりたいこと・座学的知識など）が混在し、肥大化・ごちゃつきを生む | 観点ごとにメモを分ける/まとめる仕組みがない |

## 2. 設計コンセプト

1. シンプルなメモアプリだが、**ユーザー体験(UX)を最も重視する**
2. **構造の自由度を最優先する**：階層・分類を後から自由に増減できること
3. **操作の手数を最小化する**：構造の自由度と並ぶ核の原則。入力・修正・移動などすべての操作に適用される
4. 見た目のカスタマイズは優先度を下げる（プリセットから選ぶ形に留める）
5. **構造化を強制しない**：1つのメモの中で、書きながら必要な部分だけを後から構造化できる（後述4章）

## 3. データモデル

### 3-1. メモ間の階層構造（フォルダ的なもの）
- フォルダの中にフォルダ、という再帰構造で、深さに上限を設けない（親への参照のみで実現可能）
- 実装は段階的に：①基本操作（作る・名前変更・移動）を先に作り、②一括移動・並べ替え等は後回し

### 3-1-1. フォルダ内の表示順（確定）

同じフォルダの中の一覧表示は以下のルールで並べる。

| 種別 | 並び順 |
|---|---|
| 📁 フォルダ | 五十音順（昇順）。常にメモより上に表示 |
| 📄 メモ | 最終更新日が新しい順（降順）。フォルダの下に表示 |

### 3-2. タグ（汎用的な多対多の関係）
- フォルダ・メモ・段落のいずれにも付けられる、住所とは独立した横断検索用のタグ
- 実装イメージ：`tags`テーブル（タグ一覧）と`tag_assignments`テーブル（`tag_id` + `target_type`(folder/note/paragraph) + `target_id`）の組み合わせ
- ※「ラベル」と同じ意味で使われていた用語を「タグ」に統一する

#### タグの色
- 新規タグ作成時に、自動で重複しない色を割り当てる（内容に基づく割り当てではなく、作成順）
- デフォルトパレットは**色覚に配慮した8色程度**（Okabe-Itoパレット等の検証済みの組み合わせ）
- 設定画面で、より多くの色を使う「鮮やかパレット（13色程度）」に切り替え可能
- パレットの色数を超えてタグを作成した場合は、色の重複を許容する
- 設定画面から、個別のタグの色を手動で変更できる

### 3-3. メモ内部の構造（段落・セクション）
- メモの中はタグ付きの段落・セクションに分けられる（例：目的・手順・言葉かけ・今後やりたいこと等）
- 各段落は簡略化された1行表示。マーカー行がある場合はマーカー行そのものを表示、ない場合は先頭の文字列を抜粋表示
- タップすると全文がモーダル（吹き出し）として表示される（スタック構造、後述5章）
- タグごとに、3-2で設定した色をほんのり／部分的に表示し、一覧上でも種類が一目で分かるようにする

### 3-4. メモの自由度（構造化と自由記述の併存）
- メモの種類を「構造化メモ」「自由メモ」に分けない（B案で決定）
- 1つのメモの中で、タグ付けされた部分（構造化済み）とただの文章のまま（自由記述）が混在できる
- 作成時に種類を選ぶ手間が発生しない設計

## 4. 段落操作・入力支援

### 4-1. マーカー記法による構造化
| マーカー | 役割 | 区切りの終わり |
|---|---|---|
| ⚪️（レベル1） | 大きなセクションの開始 | 次の⚪️が出るまで（空白行は無関係） |
| 🔹（レベル2） | セクション内のミニ段落の開始 | 次の🔹、または親セクションの終わりまで |

- マーカーは**改行直後にある場合のみ**有効（本文中に出てくる同じ記号とは区別される。Markdownの見出し記号と同じ考え方）
- マーカー記号はユーザーごとにカスタマイズ可能（デフォルトは⚪️🔹を推奨。「・」のような日常文章で頻出する記号は衝突しやすいため非推奨）
- マーカーを変更しても、既存メモの記号は自動変換しない（安全重視）。将来的に一括変換機能を追加検討（要：衝突防止策とセットで実装）

### 4-2. 空白行ベースの自動分割（マーカーがない場合のフォールバック）
- マーカーのない普通の文章は、空白行を区切りとして段落を自動検出できる
- 段落編集モードでタップすると、空白行に挟まれた範囲がドラッグなしでまとめて選択される（段落化の操作の一部）
- 一括セクション分け：メモ全体の空白行をまとめて検出し、一度の操作で全部セクション化する機能（拡張フェーズ）
- 範囲の微調整：自動検出された範囲の上端・下端を「＋」「－」ボタンで1行ずつ調整（ドラッグ不要、拡張フェーズ）

### 4-3. 入力補助（マーカー入力の専用ツールバー）
- 入力中、キーボード上に専用のボタン（入力アクセサリビュー）を表示し、タップでマーカーを挿入
- 表示順は文脈に応じて優先表示を切り替える（カーソル位置に応じて次に使いそうなマーカーを手前に出す）。全マーカーは常に選択可能

### 4-4. 段落編集モードの操作要件
段落編集モードでは、以下の操作ができる必要がある：
- 段落の範囲選択（ドラッグなしで完結すること）
- 段落化（範囲をタグ付きの段落にする）
- 段落解除（段落をただの文章に戻す）
- タグ付け
- タグ解除
- 段落結合（隣り合う段落を1つにまとめる。誤判定の修正にも使う）
- 段落分離（1つの段落を複数に分ける）

※通常のテキストモードと段落編集モードの間で、それぞれ何ができる／できないかという細かい境界・画面遷移の仕様は設計フェーズで検討する

## 5. 表示・UI

### 5-1. モーダルのスタック表示
- 段落をタップすると、全文が新しいモーダルとして手前に重なって表示される（push）
- 右スワイプで一番手前の層だけが消える（pop）

### 5-2. 重なりの視認性
- 奥の層を完全に隠さず、「少し小さく・ずらして・少し暗く」表示することで重なりの数を分かりやすくする

### 5-3. インライン展開とモーダル表示の併存
- タップ＝モーダル表示、横の「＋」マーク＝メモ内に直接インライン展開、の2つの表示方法を併存させる
- 「－」でインライン展開を元に戻せる
- **階層を1段掘っても同じ表示方式を引き継ぐ**（インラインで開いた段落の中の🔹も同様にインラインで展開でき、モーダルで開いた場合は中の🔹も同様にモーダルで開く）

### 5-4. 戻る操作の統一（右スワイプ）
5-1章で決めた「モーダルを右スワイプで1枚消す（pop）」という操作を、画面遷移全体で一貫した操作として適用する。

- モーダルを右スワイプ→一番手前の段落モーダルが1枚消える（5-1章と同じ）
- メモ詳細画面を右スワイプ→そのメモを開いていたフォルダの一覧画面に戻る
- フォルダ一覧画面を右スワイプ→1つ上の親フォルダの一覧画面に戻る（ルートフォルダの場合は何も起きない）
- 既存の「＜戻る」ボタンは残し、右スワイプは同じ動作を行うショートカットとして併存する

「右スワイプ＝1段階戻る」という操作を、モーダル・メモ詳細・フォルダ階層のすべての階層に共通の操作にすることで、操作の手数を減らす（2章の設計コンセプトとの整合）。

⚠️ 実装方法の詳細（画面端からのスワイプに限定するか、画面のどこからでもスワイプを開始できるか）は未確定。基本設計の詳細化時に決める。

## 6. オンボーディング（初回起動時の説明）

- このアプリの操作方法（マーカー記法・モーダル/インライン展開・タグ付け等）は独特なため、初回起動時にデモ形式で説明する
- 説明の流れの中で、初期設定（配色パレットの選択など）をその場で選ばせる
- オンボーディングは初回限定ではなく、**設定画面から後からいつでも再度見返せる**
- オンボーディング中に操作の説明用として作成されるデモ用のメモは、終了時に削除してもよいし、ユーザーが希望すればそのまま残せる（自動削除しない、選択をユーザーに委ねる）

## 7. 検索機能

### 7-1. 検索対象の範囲
| 範囲 | 結果の表示 |
|---|---|
| メモ単位 | 該当メモの一覧 |
| 段落・セクション単位 | 該当段落のみを①一覧表示 または ②結合表示 |

### 7-2. 検索条件
キーワードのみ／タグのみ／キーワード＋タグ、の組み合わせをすべて指定可能

### 7-3. 検索結果の閲覧（コマンドクエリ分離の原則）
- 検索結果の「中身を見る」操作は完全に副作用のない読み取り専用
- 「道順を確認する」「ジャンプする」は、明示的に選んだときだけ発生する別操作として完全に分離する

### 7-4. マーカー変更と検索の整合性
- マーカー記号が異なる新旧メモが混在しても、パース済みのデータ（タグ・内容）を検索するため影響はない

## 8. 技術選定

### 8-1. データベース
- **RDB（リレーショナルデータベース）を採用**
- 理由：フォルダ・メモ・タグ・段落という実体が明確で、タグ・段落単位での横断検索が要件の中心のため

### 8-2. 検索パフォーマンス対策
- 階層・タグの紐付けで使う列（`parent_id`・`tag_id`・`target_id`・`target_type`等）にインデックスを張る
- メモ本文のキーワード検索には、SQLiteのFTS5（全文検索用の機能）を採用する
- 個人利用規模のデータ量では、上記2点の対応で十分。データ量が大幅に増えた場合のみ、追加の最適化（キャッシュ等）を検討する

### 8-3. 対象プラットフォーム・開発フレームワーク
- まずiPhone（iOS）で使えるようにし、将来的にAndroidにも対応する
- データモデル・検索ロジック等のコア部分はOSに依存しないため、両OS対応による影響は小さい
- フレームワークは**React Native（Expo）**を採用する。理由：別プロジェクトのWebアプリ開発でReactを使う予定があり、言語・考え方（コンポーネント・hooks）を共有できるため
- キーボード上の専用ツールバーやスワイプ操作のような一部のOS特有機能は、既存のネイティブモジュール（npmパッケージ）を活用し、必要な場合のみ最小限のネイティブコード（Swift/Kotlin）を追加する

## 9. マネタイズ方針

### 9-1. 判断基準
**「アプリの基本的な使い方として必須な機能は無料、なくても本来の用途は果たせるが、あると便利な機能は有料」**という基準で線引きする。「全ユーザーが基本的に無料で使える」という目標を損なわないようにする。

### 9-2. 無料/有料機能一覧
| 区分 | 機能 |
|---|---|
| **無料** | メモ作成・階層構造・タグ付け（横断検索）・段落のマーカー構造化・モーダル/インライン表示・段落編集モードの基本操作（範囲選択・段落化・段落解除・タグ付け/解除・段落結合・段落分離）・メモ単位/段落単位検索・iPhoneメモ等からのインポート・操作の取り消し（Undo）・手動エクスポート（Googleドライブ等への自分でのバックアップ）・オンボーディング |
| **有料** | ①アプリ自体のロック機能（パスコード/指紋認証）<br>②自動クラウドバックアップ（継続的・自動・いつでも復元）<br>③一括セクション分け・範囲の微調整等の拡張的な便利機能<br>④鮮やかな配色パレット等、見た目の追加カスタマイズ |

## 12. 用語集

このプロジェクト特有の言葉、または一般的な言葉だがこのプロジェクトでは特定の意味で使っている言葉をまとめる。

| 用語 | 意味 |
|---|---|
| **メモ** | ユーザーが作成する1つの記録単位。内部に複数の段落を持てる |
| **フォルダ** | メモを整理するための入れ物。フォルダの中にフォルダを作れる（深さの制限なし） |
| **タグ** | フォルダ・メモ・段落のいずれにも付けられる、横断検索用のラベル。1つの対象に複数付けられる。住所（後述）とは独立した分類軸 |
| **段落（セクション）** | メモの内容を分割した、ラベル（タグ）付きの一区切り。さらに内側に小さな段落（ミニ段落）を持てる |
| **住所** | あるメモ・段落が、階層構造の中でどこに位置するか（親フォルダ・親段落の連なり）を指す呼び方。タグとは違い、1つしか持てない |
| **マーカー** | 段落の開始位置を示すための記号（デフォルトは大きな段落用の⚪️、小さな段落用の🔹）。改行の直後にある場合のみ有効。ユーザーごとに記号を変更できる |
| **段落編集モード** | 段落の範囲選択・段落化・段落解除・タグ付け/解除・段落結合・段落分離を行うための操作モード（通常の文章入力モードとは別）。空白行で区切られた範囲をドラッグなしでタップして選択する操作も、この中の「範囲選択・段落化」にあたる |
| **モーダル** | 段落をタップしたときに、画面の手前に重なって表示される全文表示の画面。タップするたびに新しいモーダルが手前に積まれ（push）、右スワイプで一番手前の1枚だけが消える（pop） |
| **インライン展開** | モーダルの代わりに、メモの中に直接（別画面を開かずに）段落の全文を展開して表示する方式。「＋」で展開、「－」で元に戻す |
| **コマンドクエリ分離の原則** | 「内容を見るだけの操作」には絶対に副作用（画面移動等）を持たせず、「移動・変更を伴う操作」は別の明示的な操作として完全に分離するという、このアプリ全体を通した設計方針 |
| **道順** | 検索結果として表示された段落が、階層構造（住所）上のどこにあるかを示す経路情報。検索結果を見るだけでは表示されず、明示的に確認操作をしたときだけ表示される |
| **オンボーディング** | 初回起動時に行う、操作方法のデモ説明と初期設定の選択。設定画面から後でも見返せる |
| **必須・拡張・将来検討** | 機能の優先順位の分類。必須＝最初のバージョンに含める、拡張＝後のバージョンで追加、将来検討＝実現するかどうか未定だが検討対象として残している機能 |
| **対象外** | 「将来検討」とは異なり、今回の開発では検討すらしないと決めた機能。後から同じ提案が出たときに再検討せず即答できるようにするための分類 |

## 13. 非機能要件

### 13-1. セキュリティ
- 前提：無料範囲はクラウド保存なし・ローカル保存のみ。クラウド連携を伴う有料機能（自動クラウドバックアップ等）のセキュリティはここでは対象外とし、別途検討する
- 対応方針：DBファイルにiOSの標準ファイル保護属性（`NSFileProtectionComplete`、端末のパスコード/Face ID由来の鍵で自動的に保護される仕組み）を設定する。アプリ独自のパスワードや暗号化処理は実装しない
- 通信を行わない（ネットワークAPIを持たない）ため、通信経路上の脆弱性は対象外
- 検索条件のSQL組み立てはパラメータ化クエリを用いる（実装時の注意点。詳細は詳細設計で具体化）

### 13-2. 可用性
- サーバーを持たないローカル専用アプリのため、稼働率・ダウンタイムという意味での可用性は本要件定義では対象外とする
- 代わりに、データの耐障害性として以下を要件とする
  - メモ・段落の変更内容はリアルタイム（都度）保存する。アプリが強制終了されても、直前の保存タイミングまでの内容は失われないようにする
  - DBファイル自体が破損した場合の自動復旧は保証しない（既知の制約として明記する）

### 13-3. 保守性
- DBスキーマのマイグレーション戦略が必要（将来のカラム追加等に対応するため）。詳細は実装時に具体化する
- UI層とデータアクセス層を分離する。実装方法（関数・モジュール単位の分離等）は実装時に具体化する

### 13-4. 補足：OS標準バックアップと有料の自動クラウドバックアップ機能の違い
- iOS標準のiCloud/ローカルバックアップには、本アプリのDBファイルも自動的に含まれる（アプリ側で特別な実装は不要）
- ただし以下の限界があるため、有料の「自動クラウドバックアップ」機能（9-2章・将来検討）には別の価値がある
  - 端末全体の丸ごと復元のみで、メモだけを選んで特定時点に戻すことはできない
  - バックアップのタイミングはOSが管理し、アプリ側から制御・保証できない
  - ユーザーがiCloudバックアップ自体をOFFにしている場合は機能しない
  - 将来Android対応する場合、iCloudの仕組みは使えないため、独自バックアップがあればプラットフォームを問わず同じ安心感を提供できる
