# 課題管理表

設計レビューで洗い出された課題を管理します。

- **ステータス**: `未対応` / `対応中` / `対応済み` / `対応不要`
- **発見元**: `設計レビュー（2026-06-25）`

---

## 優先度：高（テーブル定義書着手前に解決必須）

| # | 課題 | 影響する成果物 | ステータス | 対応内容・メモ |
|---|------|----------------|------------|----------------|
| H-1 | メモは必ずフォルダに属するか（`NOTE.folder_id` はNOT NULLか） | テーブル定義書 | 対応済み | **`folder_id` は NOT NULL**。アプリ初回起動時にデフォルトフォルダを自動生成し、メモは必ずいずれかのフォルダに属する。ユーザーが整理する意識を持って使うことを前提とするため、フォルダ未所属のメモは想定しない。 |
| H-2 | `PARAGRAPH`テーブルの保持方式：全段落が`note_id`を持つか（パターンB）、トップレベルのみ持つか（パターンA） | テーブル定義書・パーサ設計 | 対応済み | **パターンB採用**（全段落が`note_id` NOT NULLを持つ）。`parent_paragraph_id`はトップレベルがNULL、ネスト段落は親のIDを持つ。表示クエリが単純（`WHERE note_id = ?`）になる利点を優先。段落移動時の子孫一括更新コストは実装時に対処。 |
| H-3 | 段落の表示順の管理方式：DBに`order`カラムを持つか、本文テキストのパース結果として順序を導出するか | テーブル定義書・メモ詳細設計書 | 対応済み | **DBに`order`カラムを持つ方式**。プレーンテキストも`type='text'`の段落として扱い、PARAGRAPHテーブルに格納する。`type='paragraph'`は折りたたみ可能な段落、`type='text'`は通常テキスト行。`order`は「自分の親要素の中での順序」を表す。タイプによって列の使用有無（タイトルの有無等）が変わる詳細は詳細設計で確定。 |
| H-4 | ゴミ箱のデータモデル：ソフトデリート（`deleted_at`カラム追加）か別テーブル方式か | テーブル定義書 | 対応済み | **`NOTE`テーブルに`deleted_at TIMESTAMP NULL`カラムを追加するソフトデリート方式**。`folder_id`は削除時も保持し、復元時は元のフォルダにそのまま戻る。booleanフラグではなくタイムスタンプにすることで、削除日時の記録・将来的な自動完全削除（30日後など）にも対応できる。 |
